# راهنمای مدیریت هوش مصنوعی (ادمین و کاربر)

## نوع ارائه‌دهنده از کجا پر می‌شود؟

- در فرانت، لیست کشویی از **`GET /api/admin/ai-provider-templates`** پر می‌شود؛ داده از جدول **`tbl_ai_provider_templates`** می‌آید (مهاجرت [`005_ai_provider_templates.sql`](../backend/report_backend/migrations/005_ai_provider_templates.sql)).
- در بک‌اند، مقدار `provider_type` همان **slug** قالب است؛ در [`aiProviderTemplateService.js`](../backend/report_backend/src/services/aiProviderTemplateService.js) با قالب ادغام می‌شود و در [`llmInvoke.js`](../backend/report_backend/src/services/llmInvoke.js) بر اساس **engine** قالب (مثلاً `google_gemini` یا `openai_chat`) فراخوانی انجام می‌شود.
- اعتبارسنجی طول و فرمت slug در [`promptFieldLimits.js`](../backend/report_backend/src/constants/promptFieldLimits.js) است؛ وجود slug در رجیستری و فعال بودن آن با **`assertProviderSlugAllowed`** هنگام ذخیرهٔ ردیف API بررسی می‌شود.

## کاربرد (`usage_key`) یعنی چه؟

- یک **شناسه ثابت** در کد است که می‌گوید این ردیف برای **کدام ویژگی برنامه** است (مثلاً تولید خلاصه مدیریتی).
- مقدار **`field.management_summary`** یعنی: «تمام ردیف‌هایی که این کاربرد را دارند، برای فراخوانی مدل در **خلاصه مدیریتی گزارشات میدانی** استفاده می‌شوند».
- چند ردیف با **یکسان بودن `usage_key`** و **`sort_order` متفاوت** = اولویت و **پشتیبان (fallback)** اگر مدل اول خطا بدهد.

## دکمه «بروزرسانی لیست از سرور»

- درخواست **`GET /api/admin/ai-api-configs`** را دوباره می‌زند و جدول را از دیتابیس پر می‌کند؛ فیلترهای بالای صفحه را عوض نمی‌کند (مگر اینکه خودتان فیلتر را عوض کرده باشید).

## جریان کار

1. **ادمین** در «مدیریت API» اتصال به سرویس را با انتخاب **قالب** (slug)، مدل و کلید (ترجیحاً env) ثبت می‌کند؛ برای سرویس‌های سازگار با OpenAI معمولاً قالب `openai_chat` یا `avalai` و فیلد JSON `base_url` کافی است.
2. **ادمین** در «مدیریت پرامپت‌ها» متن دستورالعمل‌ها را ویرایش می‌کند.
3. **ادمین** (در صورت نیاز) در «اکشن‌های AI فرم‌ها» برای هر دکمه، **فرم و اکشن را از لیست ثبت‌شده در کد** انتخاب می‌کند، **روش ساخت متن** را مشخص می‌کند، در حالت ساده **پرامپت را از لیست رجیستری** برمی‌گزیند، و **کاربرد API** (`usage_key`) و در صورت نیاز **اولویت ردیف** (`ai_config_id`) را تنظیم می‌کند.
4. **کاربر مجاز** در فرم (مثلاً ایجاد خلاصه مدیریتی) فقط دکمه تولید را می‌زند؛ سرور بر اساس کانفیگ، مدل را صدا می‌زند و متن را برمی‌گرداند — بدون دیدن API key.

## اکشن‌های AI فرم‌ها — واژه‌ها به زبان ساده

- **فرم و نام اکشن** باید در **رجیستری برنامه** باشند ([`aiFormRegistry.js`](../backend/report_backend/src/constants/aiFormRegistry.js) در بک‌اند و [`REGISTERED_FORM_AI_ACTIONS`](../frontend/src/constants/aiFormNames.js) در فرانت)؛ راهبر فقط از کشو انتخاب می‌کند و نام انگلیسی تایپ نمی‌کند. افزودن فرم/دکمهٔ جدید با توسعه‌دهنده است.
- **`usage_key` در این صفحه** همان **کاربرد API** است که در «مدیریت API هوش» برای هر ردیف نوشته‌اید؛ همهٔ ردیف‌های فعال با این مقدار، به ترتیب اولویت، برای تماس با مدل امتحان می‌شوند (اگر یکی خطا بدهد، بعدی).
- **`ai_config_id` (اختیاری)** شناسهٔ **یک ردیف مشخص** در همان جدول API است؛ اگر انتخاب شود، سرور **اول** همان ردیف را امتحان می‌کند، بعد در صورت نیاز بقیهٔ زنجیرهٔ همان `usage_key` را.
- **`labeled_fields`** / **`field_management_summary_v1`** در ردیف‌های قدیمی ممکن است دیده شود؛ معادل جدید **`unified_v1`** است: یک مسیر، با پرامپت ساده از رجیستری + دادهٔ فرم، یا برای خلاصه میدانی ساخت متن در سرور.

## OpenAI / سرویس سازگار

- نوع **`openai_chat`**: درخواست به endpoint چت سازگار با OpenAI (`.../v1/chat/completions`) با هدر `Authorization: Bearer <کلید>`.
- در JSON اضافی می‌توان `base_url` یا `chat_completions_url` را گذاشت (جزئیات در راهنمای داخل برنامه و فایل `llmInvoke.js`).

## مانده اعتبار حساب (راهبر — بدون SQL و بدون تغییر کد)

برای دیدن باقی‌مانده اعتبار سرویس‌هایی که API balance دارند:

1. منو → **مدیریت API هوش مصنوعی**
2. ردیف مورد نظر را **ویرایش** کنید (یا ردیف جدید بسازید)
3. در بخش **«مانده اعتبار حساب (دکمه کیف پول در جدول)»** تیک بزنید
4. فیلدها را از مستندات همان سرویس پر کنید:

| فیلد در فرم | معنی |
|-------------|------|
| آدرس API مانده | URL کامل GET (مثلاً `https://api.avalai.ir/user/v1/credit`) |
| نام فیلد مانده در JSON | کلید پاسخ که عدد مانده است (مثلاً `remaining_unit`) |
| فیلد ثانویه (اختیاری) | مانده دیگر (مثلاً `remaining_irt` برای ریال) |
| برچسب واحد | فقط نمایش (مثلاً `UNIT`) |

5. **ذخیره** → در جدول دکمه **کیف پول** ظاهر می‌شود → مانده در ستون «مانده» نمایش داده می‌شود

**نکات:**

- کلید API همان اعتبارنامه همین ردیف است (متغیر env یا secret ذخیره‌شده)
- درخواست از **سرور** انجام می‌شود؛ کلید در مرورگر لو نمی‌رود
- اگر نوع ارائه‌دهنده **avalai** باشد، با انتخاب آن نوع فیلدها از قالب پیش‌فرض پر می‌شوند
- تنظیمات مانده در `extra_config.credit_check` همان ردیف ذخیره می‌شود — نیازی به ویرایش JSON دستی نیست (بخش JSON دیگر `credit_check` را نشان نمی‌دهد)

### مثال AvalAI

```
URL:     https://api.avalai.ir/user/v1/credit
فیلد:    remaining_unit
ثانویه:  remaining_irt
واحد:    UNIT
کلید:    AVALAI_API_KEY در .env سرور
```

مستندات: [AvalAI User API](https://docs.avalai.ir/en/api-reference/user)

### سرویس جدید

1. از مستندات سرویس آدرس «مانده / balance / credit» را بگیرید
2. یک بار با همان API key تست کنید و نام فیلد JSON پاسخ را یادداشت کنید
3. در فرم ردیف API همان مقادیر را وارد و ذخیره کنید

اگر سرویس API مانده ندارد، تیک بخش مانده را بردارید.
